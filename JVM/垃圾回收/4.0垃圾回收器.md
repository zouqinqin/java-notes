## 什么是垃圾回收器？

> 🧹 **通俗理解**：垃圾回收器就是 JVM 的"清洁工"。程序运行时会不断创建对象（产生垃圾），垃圾回收器负责定期清理这些没用的对象，释放内存空间。
>
> 不同的"清洁工"有不同的工作方式：有的是一个人慢慢打扫（串行），有的是多人同时打扫（并行），有的是边打扫边让你继续工作（并发）。

**核心矛盾**：垃圾回收时需要暂停用户线程（STW，Stop The World），就像打扫房间时让你先别动东西。我们希望这个暂停时间越短越好。

---

1. 串行
   - 单线程
   - 堆内存较小，适合个人电脑
2. 吞吐量优先
   - 多线程
   - 堆内存较大，多核CPU
   - 让单位时间内，STW的时间最短（总时间）
3. 响应时间优先
   - 多线程
   - 堆内存较大，多核CPU
   - 尽可能让单次STW的时间最短（每次垃圾回收时间短）

### 4.1 串行（Serial）

- `-XX:+UseSerialGC` = Serial + SerialOld

**实现说明：**
- 垃圾回收时，所有用户线程暂停（STW），由单个 GC 线程完成回收
- 新生代使用 Serial 收集器，采用复制算法，将存活对象从 Eden/From 区复制到 To 区
- 老年代使用 SerialOld 收集器，采用标记整理算法，避免内存碎片
- 简单高效，适合堆内存较小（几十到几百 MB）的单核或低配环境

### 4.2 吞吐量优先（Parallel）

**参数说明：**

| 参数 | 说明 |
|------|------|
| `-XX:+UseParallelGC` ~ `-XX:+UseParallelOldGC` | 开启并行 GC，两者互相激活。新生代使用 ParallelGC（复制算法），老年代使用 ParallelOldGC（标记整理算法） |
| `-XX:+UseAdaptiveSizePolicy` | 自适应大小策略，JVM 根据运行时数据自动调整 Eden、Survivor 区比例，无需手动设置 `-Xmn`、`-XX:SurvivorRatio` 等参数 |
| `-XX:GCTimeRatio=ratio` | 设置 GC 时间占总运行时间的比例目标，公式为 `1/(1+ratio)`。默认 ratio=99，即 GC 时间不超过 1%；设置为 19 时 GC 时间不超过 5%。JVM 会通过扩大堆来满足此目标 |
| `-XX:MaxGCPauseMillis=ms` | 设置每次 GC 最大暂停时间目标，默认 200ms。JVM 会通过缩小堆来满足此目标，可能降低吞吐量 |
| `-XX:ParallelGCThreads=n` | 设置 STW 阶段并行 GC 的线程数，默认与 CPU 核心数相同 |

> 注：`GCTimeRatio` 和 `MaxGCPauseMillis` 目标相互冲突——前者希望堆大（减少 GC 频率），后者希望堆小（缩短单次暂停）。通常只设置其中一个。

**实现说明：**
- 多个 GC 线程并行工作，STW 期间同时回收，缩短总 GC 时间，提升吞吐量
- 新生代 ParallelGC 与 Serial 类似，但多线程并行执行复制算法
- 老年代 ParallelOldGC 多线程并行执行标记整理算法
- 适合后台计算型任务（批处理、数据分析），对单次暂停时间不敏感

### 4.3 响应时间优先（CMS）

**参数说明：**

| 参数 | 说明 |
|------|------|
| `-XX:+UseConcMarkSweepGC` ~ `-XX:+UseParNewGC` ~ `SerialOld` | 开启 CMS 收集器。新生代使用 ParNew（并行复制算法），老年代使用 CMS（并发标记清除算法）；当 CMS 出现并发模式失败时，自动退化为 SerialOld（单线程标记整理） |
| `-XX:ParallelGCThreads=n` ~ `-XX:ConcGCThreads=threads` | `ParallelGCThreads` 设置 STW 阶段（初始标记、重新标记）的并行线程数；`ConcGCThreads` 设置并发阶段（并发标记、并发清除）的线程数，一般设为 `ParallelGCThreads` 的 1/4，避免占用过多 CPU |
| `-XX:CMSInitiatingOccupancyFraction=percent` | 设置老年代内存占用达到多少百分比时触发 CMS GC，默认约 68%。设置过高可能导致 CMS 来不及回收，触发并发模式失败，退化为 SerialOld；设置过低则 GC 过于频繁 |
| `-XX:+CMSScavengeBeforeRemark` | 在重新标记（Remark）阶段前，先执行一次 Minor GC，清理新生代中的短命对象，减少 Remark 阶段需要跨代扫描的对象数量，从而缩短 Remark 的 STW 时间 |

**实现说明（CMS 四阶段）：**

1. **初始标记（Initial Mark）**— STW，仅标记 GC Roots 直接关联的对象，速度极快
2. **并发标记（Concurrent Mark）**— 与用户线程并发执行，从 GC Roots 出发遍历整个对象图，耗时最长但不停顿
3. **重新标记（Remark）**— STW，修正并发标记期间因用户线程运行导致的标记变动（增量更新），时间较短
4. **并发清除（Concurrent Sweep）**— 与用户线程并发执行，清除未标记对象，采用标记清除算法（不整理内存，存在碎片）

> 缺点：使用标记清除算法会产生内存碎片；并发阶段占用 CPU 资源；若老年代在并发清除期间被填满，会触发"并发模式失败"，退化为 SerialOld，造成较长 STW。

### 4.4 G1（Garbage First）

> 🏘️ **通俗理解**：G1 把堆内存切成很多小格子（Region），每次优先清理垃圾最多的格子，就像先打扫最脏的房间，既能控制打扫时间，又能保证效果。

**启用参数：**
```
-XX:+UseG1GC
```

**核心设计：**

| 特性 | 说明 |
|------|------|
| Region 分区 | 将堆划分为 2048 个大小相等的 Region（1~32MB），每个 Region 动态扮演 Eden/Survivor/Old 角色 |
| 优先回收 | 优先选择垃圾比例最高的 Region 回收（Garbage First 名字由来） |
| 可预测停顿 | 通过 `-XX:MaxGCPauseMillis=ms`（默认 200ms）控制每次 GC 的最大停顿时间 |
| 无碎片 | 使用复制算法在 Region 间移动对象，不产生内存碎片 |

**G1 回收阶段：**

1. **新生代回收（Young GC）**— STW，多线程并行，回收 Eden 和 Survivor Region
2. **并发标记（Concurrent Marking）**— 与用户线程并发，标记老年代存活对象
3. **混合回收（Mixed GC）**— STW，同时回收新生代 + 垃圾最多的老年代 Region
4. **Full GC（兜底）**— 内存实在不够时触发，单线程标记整理，尽量避免

**常用参数：**

| 参数 | 说明 |
|------|------|
| `-XX:MaxGCPauseMillis=200` | 目标最大停顿时间，G1 会自动调整回收范围来满足 |
| `-XX:+UseG1GC` | 开启 G1（JDK 9+ 默认） |
| `-XX:G1HeapRegionSize=n` | 设置每个 Region 大小（1~32MB，必须是 2 的幂） |
| `-XX:G1NewSizePercent=5` | 新生代最小占堆比例，默认 5% |
| `-XX:G1MaxNewSizePercent=60` | 新生代最大占堆比例，默认 60% |

> ✅ **适用场景**：大堆内存（4GB 以上）、需要可预测低延迟的服务端应用。JDK 9 起成为默认收集器，替代 CMS。

### 4.5 ZGC（Z Garbage Collector）

> ⚡ **通俗理解**：ZGC 是目前最先进的"清洁工"，几乎全程不打扰你工作（并发），停顿时间控制在 **10ms 以内**，不管堆有多大都能做到。

**启用参数：**
```
-XX:+UseZGC          # JDK 15+ 正式可用
```

**核心特点：**

| 特性 | 说明 |
|------|------|
| 超低停顿 | STW 时间 < 10ms，与堆大小无关（TB 级堆也能做到） |
| 全并发 | 标记、转移、重定位全部与用户线程并发执行 |
| 染色指针 | 利用 64 位指针的高位存储 GC 状态信息，无需额外内存 |
| 读屏障 | 通过读屏障（Load Barrier）实现并发移动对象时的引用修正 |
| 无分代（JDK 21前） | 早期版本不区分新老年代，JDK 21 引入分代 ZGC 进一步提升性能 |

**适用场景：**
- 对延迟极度敏感的应用（金融交易、实时系统）
- 超大堆内存（几十 GB 甚至 TB 级）
- JDK 15+ 生产环境

> ⚠️ **注意**：ZGC 吞吐量略低于 G1（并发 GC 线程会占用 CPU），如果追求最大吞吐量仍推荐 Parallel GC。

## 收集器横向对比

| 收集器 | 算法 | 线程模式 | 停顿时间 | 吞吐量 | 适用场景 | JDK 版本 |
|--------|------|----------|----------|--------|----------|----------|
| Serial | 复制 + 标记整理 | 单线程 | 长 | 低 | 客户端、小堆 | 全版本 |
| Parallel | 复制 + 标记整理 | 多线程并行 | 中 | ⭐⭐⭐ 高 | 批处理、后台计算 | 全版本 |
| CMS | 标记清除 | 并发 | 短 | 中 | 低延迟服务端（已废弃） | JDK 9 废弃 |
| G1 | 复制 + 标记整理 | 并发 + 并行 | 可控 | ⭐⭐ 中高 | 大堆、通用服务端 | JDK 9 默认 |
| ZGC | 并发复制 | 全并发 | ⭐ 极短(<10ms) | 中 | 超大堆、极低延迟 | JDK 15+ |

**一句话选型建议：**
- 🖥️ 客户端 / 小内存 → **Serial**
- ⚙️ 后台批处理 / 追求吞吐量 → **Parallel GC**
- 🌐 普通服务端（JDK 9+）→ **G1**（默认，无需配置）
- 🚀 超低延迟 / 大堆（JDK 15+）→ **ZGC**
