### 双亲委派
- 收到类加载请求时，先不自己加载，而是委托给父加载器，父加载器再委托给它的父加载器，一直向上到 Bootstrap。父加载器找不到这个类，才往下交给子加载器自己加载。
- 关键是先向上委托，上面找不到再向下自己加载，不是从上往下主动找。


### Tomcat 破坏双亲委派的原因是：

1. Tomcat 要同时部署多个 Web 应用，每个应用可能依赖同一个库的不同版本（比如应用A用 Spring 4，应用B用 Spring 5）。如果用双亲委派，同一个类名只能加载一个版本，多个应用就会互相干扰。所以 Tomcat 给每个 Web 应用单独配了一个 WebAppClassLoader，优先加载自己 WEB-INF/lib 下的类，不先委托父加载器。

- 为什么说重写 findClass 而不是 loadClass
- 注释最后一句话就是答案：

- Subclasses are encouraged to override findClass, rather than this method.

1. 因为 loadClass 里写的就是双亲委派的逻辑，你如果重写它就把委派逻辑覆盖掉了（破坏双亲委派）。
2. 只重写 findClass 的话，委派逻辑还在，只是告诉 JVM "找不到的时候来问我"。

```text
双亲委派机制
    ↓ 是什么
    loadClass() 先委托父加载器，找不到才自己加载

    ↓ 怎么打破
    重写 loadClass()，让特定包名的类优先自己加载

    ↓ 打破有什么用
    热部署：新 ClassLoader 实例 = 空缓存 = 强制读最新 .class
    Tomcat：每个应用独立 ClassLoader = 多版本隔离

```

- **先理解 JVM 的类唯一性规则**

1. JVM 判断两个类是否是"同一个类"，条件是： 类全名相同  +  加载它的 ClassLoader 实例相同 = 同一个类

### 代码案例

#### 1. 遵守双亲委派 —— 只重写 `findClass`

```java
public class NormalClassLoader extends ClassLoader {

    private String classPath;

    public NormalClassLoader(String classPath) {
        this.classPath = classPath;
    }

    /**
     * 只重写 findClass，不动 loadClass
     * 双亲委派逻辑完全保留
     * 父加载器找不到时，才会调用这个方法
     */
    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        String filePath = classPath + name.replace(".", "/") + ".class";
        try {
            byte[] bytes = Files.readAllBytes(Paths.get(filePath));
            return defineClass(name, bytes, 0, bytes.length);
        } catch (IOException e) {
            throw new ClassNotFoundException("找不到类文件: " + filePath);
        }
    }
}
```

> `loadClass()` 里的委派逻辑不动，父加载器找不到时才回调 `findClass()`，这是推荐写法。

---

#### 2. 打破双亲委派 —— 重写 `loadClass`（热部署场景）

```java
public class BreakParentDelegation extends ClassLoader {

    @Override
    protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
        synchronized (getClassLoadingLock(name)) {
            Class<?> c = findLoadedClass(name);
            if (c == null) {
                // java. 开头的核心类必须交给父加载器，否则抛 SecurityException
                if (name.startsWith("java.")) {
                    c = getParent().loadClass(name);
                } else {
                    // 先自己加载，找不到再委托父加载器
                    try {
                        c = findClass(name);
                    } catch (ClassNotFoundException e) {
                        c = getParent().loadClass(name);
                    }
                }
            }
            if (resolve) resolveClass(c);
            return c;
        }
    }
}
```

> 热部署原理：每次创建一个**新的 ClassLoader 实例**，缓存为空，强制重新读取最新的 `.class` 文件。

---

#### 3. 模拟 Tomcat WebAppClassLoader（多应用隔离）

```java
public class WebAppClassLoader extends ClassLoader {

    private String appId;
    private String webInfLibPath;

    public WebAppClassLoader(String appId, String webInfLibPath) {
        super(null); // 父加载器设为 null，直接用 Bootstrap
        this.appId = appId;
        this.webInfLibPath = webInfLibPath;
    }

    @Override
    protected Class<?> loadClass(String name, boolean resolve) throws ClassNotFoundException {
        synchronized (getClassLoadingLock(name)) {
            // java./javax. 必须走系统类加载器
            if (name.startsWith("java.") || name.startsWith("javax.")) {
                return ClassLoader.getSystemClassLoader().loadClass(name);
            }
            // 优先从自己的 WEB-INF/lib 加载（打破双亲委派的核心）
            try {
                Class<?> c = findClass(name);
                if (resolve) resolveClass(c);
                return c;
            } catch (ClassNotFoundException e) {
                // 找不到再委托系统类加载器
                return ClassLoader.getSystemClassLoader().loadClass(name);
            }
        }
    }
}
```

> 应用A 和 应用B 各自持有一个 `WebAppClassLoader` 实例，加载的 `Spring` 类虽然全名相同，但 ClassLoader 实例不同，JVM 视为两个不同的类，互不干扰。

---

### 三种方式对比

| 方式 | 重写方法 | 是否打破委派 | 典型场景 |
|------|----------|-------------|----------|
| `NormalClassLoader` | `findClass` | ❌ 否 | 自定义路径加载，委派保留 |
| `BreakParentDelegation` | `loadClass` | ✅ 是 | 热部署，强制加载最新 .class |
| `WebAppClassLoader` | `loadClass` | ✅ 是 | Tomcat 多应用版本隔离 |
